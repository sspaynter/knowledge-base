# v2.2.1 — Vault Ownership Guard

**Released:** 2026-03-19
**Session:** 79
**Sprint:** infra-backend-ops

## Changes

### Fixed
- Vault directory ownership guard — container entrypoint now runs `chown -R app:app` on `/app/vault` and `/app/uploads` before dropping privileges, preventing write failures when volume-mounted directories have host-side UIDs

### Changed
- Dockerfile uses `ENTRYPOINT` with `su-exec` privilege drop instead of `USER app` + `CMD`
- Added `su-exec` Alpine package to container image

## Context

The `products/ssa-website/` vault directory on the NAS had UID 501:20 (macOS user) ownership instead of UID 100:101 (container `app` user). The container could read but not write to this directory, causing HTTP 500 errors on `kb-sync.sh` writes.

Root cause: a process wrote directly to the NAS filesystem from macOS, bypassing the KB API. The API writes as the container's `app` user (correct); direct filesystem writes preserve the source UID (incorrect).

Full incident report: [vault-ownership-incident-2026-03-19](/page/operations/infrastructure/vault-ownership-incident-2026-03-19)
