# Applyr — Cover Letter Generation Architecture

## Overview

Cover letters are generated by calling the Claude API from `server/services/coverLetter.js`. The system prompt is assembled from four independent layers. Each layer has a clear owner, a defined token budget, and changes independently.

This architecture supports multi-tenancy: new users get Layers 1 and 2 for free and only need to build their voice profile (Layer 3).

---

## 4-Layer Prompt Architecture

```
Layer 1: Craft (universal, hardcoded)      ~200 tokens
    +
Layer 2: Quality (universal, hardcoded)    ~100 tokens
    +
Layer 3: Voice (per-user, from DB)         ~200-300 tokens
    =
System prompt sent to Claude API           ~500-600 tokens

Layer 4: Job Context (per-generation)       variable
    =
User message sent to Claude API
```

### Layer 1: Cover Letter Craft

The skill of writing a good cover letter. This is the product's expertise.

**Opening:**
- First-person structural constraint: the first sentence after the salutation must be from the applicant's perspective, not a third-person statement about the company
- Connect a specific company detail to direct personal experience — show recognition, not research
- End the opening paragraph by naming the hard challenge ahead, framed with respect for the people already working on it
- 3 anonymised example openings in `<examples>` tags demonstrate the structural pattern without prescribing phrases

**Body:**
- SARR pattern for experience stories (Situation, Action, Result, Relevance)
- 2 strong stories beat 5 weak ones, each needs a quantified outcome

**Close:**
- Reinforce value proposition, direct ask, no cliche sign-offs

**Format:**
- 3-4 paragraphs, 280-400 words
- Arc: recognition of their challenge → evidence from your experience → fit for this role

**Owner:** The app (hardcoded in code)
**Changes when:** Product improvement — refining what makes a good cover letter
**Applies to:** All users

### Layer 2: Writing Quality

Universal rules that make any cover letter sound human.

**Anti-slop bans (7 categories):**
- Intensifiers: deeply, truly, fundamentally, inherently, remarkably, incredibly
- Jargon: navigate, lean into, landscape, game-changer, deep dive, holistic, robust, seamless, cutting-edge, delve, harness, spearhead
- Filler: "At its core", "In today's [X]", "When it comes to", "interesting position", "interesting challenge"
- Throat-clearing: "Here's the thing", "The truth is", "Let me be clear"
- Binary contrasts: "[X] isn't the problem. [Y] is."
- Dramatic fragments: staccato "And X. And Y."
- Hedging: "perhaps", "arguably", "one might say"

**Style directive:** Write direct, specific sentences. State the point — do not announce it.

**Owner:** The app (hardcoded in code)
**Changes when:** New anti-AI patterns discovered
**Applies to:** All users
**Source:** `~/.claude/skills/anti-slop/` (phrases.md + structures.md)

### Layer 3: Voice Profile

Personal writing instructions unique to this user. Stored in `voice_profiles.profile_data` (jsonb).

**Schema:**

```json
{
  "name": "User Name",
  "location": "City, Country",
  "voice": {
    "style_principles": ["Direct", "Reflective", "Convicted", "Vary sentence length", ...],
    "natural_phrases": ["End-to-end", "Discovery as a named phase", ...],
    "phrases_to_avoid": ["Corporate buzzwords", "Weak connectors", "Robotic transitions", ...],
    "tone_test": "Read it aloud. If it sounds like a press release, rewrite it.",
    "opening_test": "Could this opening be sent to any other company? If yes, rewrite it."
  },
  "framework": {
    "name": "Problem-Solution-Evidence",
    "logic": ["Show you recognise the challenge — from genuine understanding", "Show a parallel problem you solved", "State what you will deliver"],
    "reframes": "Shifts the reader from 'what has this person done?' to 'can this person solve my problem?'"
  },
  "structure": {
    "paragraphs": "3-4",
    "word_count": "280-380",
    "opening": "Open from genuine recognition of the company's problem. Surface personal lived experience. End on the tension.",
    "middle": "Pick 2-3 experiences closest to this role, SARR pattern, quantified outcomes.",
    "close": "Reinforce core value proposition, direct confident ask."
  },
  "differentiators": ["Design background", "AI transformation", "Internal capability building", ...],
  "ammunition": { /* themed experience library — see below */ },
  "experience_selection": {
    "pm_track": { "primary": [...], "note": "..." },
    "it_track": { "primary": [...], "note": "..." }
  }
}
```

**Owner:** Each user (stored in database)
**Changes when:** User refines their profile, harvest step promotes learnings
**Applies to:** One user

### Layer 4: Job Context

Per-generation data sent as the user message.

- Job description, company, role, location, arrangement
- Company research (overview, culture signals, interview angles)
- User notes on this role (highest priority context)
- JD highlights (sections user flagged as personally relevant)
- Resume text (truncated to 3000 chars)
- Ammunition library (selected stories, ordered by track relevance)

**Owner:** Each job application
**Changes when:** Every generation
**Applies to:** One cover letter

## Ammunition Library

The ammunition library in the voice profile organises experience stories by theme:

| Theme | Examples |
|---|---|
| `transformation` | CAPEX-to-OPEX transition, XaaS team creation |
| `leadership_teams` | 70-staff operations, team built from scratch |
| `platform_building` | Cloud X Flex, DaaS, AWS practice |
| `automation_ai` | Customer onboarding automation, patching automation |
| `vendor_commercial` | $16M vendor management, storage procurement |
| `discovery_research` | Three-phase discovery, hypothesis-driven startup |
| `stakeholder_executive` | C-suite strategy, 50+ BU engagement |

Each entry has: `title`, `employer`, `metric`, `story`.

**Selection logic:** `buildAmmunitionContext()` orders themes by track (PM track leads with platform_building/discovery; IT track leads with transformation/leadership). All themes are included but priority ordering determines which stories the model sees first and is most likely to select.

**Planned improvements (#142):** Score ammunition against JD keywords and send only top 5-8 relevant stories instead of the full library.

## Rewrite Loop

```
Generate v1 (Layers 1-4 → Claude API)
    ↓
User reviews, adds inline feedback (text selection → comment cards)
    ↓
User adds general feedback
    ↓
Rewrite (compiled feedback + existing letter → Claude API)
    ↓
New version created (immutable — v1 preserved)
    ↓
Accept or loop again
```

The rewrite prompt (`buildRewritePrompt()`) includes voice profile style principles and the quality layer anti-slop rules.

## Generation Scoring (Planned — #157)

Inspired by [Karpathy's AutoResearch](https://github.com/karpathy/autoresearch) loop pattern: the system needs a measurable proxy metric (equivalent to AutoResearch's `val_bpb`) to enable evidence-based learning instead of purely subjective review. See `operations/engineering-practice/autoresearch-learning-loop-pattern.md` for the full pattern analysis.

After each generation, a second Claude call scores the letter against a fixed rubric:

| Criterion | What it measures | Scale |
|---|---|---|
| Voice match | Does it sound like the user? | 1-5 |
| Specificity | Named projects/metrics vs generic claims | 1-5 |
| Anti-slop compliance | Banned phrases/structures detected | Pass/Fail |
| Structure adherence | SARR pattern, word count | 1-5 |
| Opening quality | First-person recognition vs third-person analysis | 1-5 |

Scores stored in `generation_scores` on the `cover_letters` table alongside: ammunition stories selected, rewrite count, user action, classified feedback themes.

This data feeds the harvest step (#145) and weekly consolidation (#146) with structured signals for what to promote, flag, or retire in the voice profile.

**AutoResearch mapping:**

| AutoResearch | Applyr equivalent |
|---|---|
| `program.md` (human strategy) | Layers 1+2 (craft + quality) |
| `train.py` (agent modifies) | Layer 3 voice profile |
| `prepare.py` (fixed evaluation) | Rubric evaluator (fixed criteria) |
| `results.tsv` (experiment log) | `generation_scores` column |
| val_bpb (single metric) | Rubric composite + rewrite count |
| keep/discard (binary) | Accept first-gen / rewrite / discard |

## Prompt Engineering Principles

See `applyr-cover-letter-prompt-engineering.md` for the full research and rationale. Key principles applied:

1. **Positive structural constraints over negative bans** — "The first sentence must be first-person" works better than "Never open with a business assessment" (Pink Elephant Problem)
2. **Intent descriptions over prescriptive phrases** — describe what the opening must achieve, not which words to use (Anthropic best practices)
3. **Examples for structural steering** — 2-3 diverse anonymised examples in `<example>` tags teach the pattern implicitly (most effective technique per Anthropic docs)
4. **Banned lists for quality gates only** — keep bans in Layer 2 (quality), not Layer 1 (craft)
