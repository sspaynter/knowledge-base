## Context

In February 2026, a session explored how people use AI (specifically Claude) for wireframing, UI mockups, and prototype review as part of the product development workflow. This article captures the research findings and the decisions made for Simon's own workflow.

---

## What the research found

### Industry has moved from wireframes to clickable prototypes

ASCII wireframes are still used as machine-readable specs inside Claude Code (e.g., `WIREFRAME.md`) to guide build precision. But for human review of flows and navigation, the industry has largely moved to clickable HTML prototypes generated by Claude.

The reason: Claude can generate a fully navigable, multi-screen HTML prototype in one pass, faster than you can sketch a wireframe.

**Clickable prototypes are genuinely interactive:**
- Buttons route to correct screens
- Nav links work
- Dropdowns, modals, and transitions all function
- User flows are completable end-to-end

### ASCII wireframes still have a role

They serve as a machine-readable UI spec passed to Claude Code during implementation. They are written *for Claude*, not for human review. The distinction matters:

| Format | Purpose | Audience |
|---|---|---|
| ASCII wireframe / `WIREFRAME.md` | Layout spec for Claude Code | Claude (build context) |
| Clickable HTML prototype | Flow and navigation review | Simon (human review) |

### Key tools/approaches observed

| Approach | What it does |
|---|---|
| Claude Artifacts (claude.ai) | Describe the app; Claude generates a clickable prototype in the Artifacts panel. Click through immediately. No files. Fast loop, no persistence. |
| Claude Code → single HTML file | Claude generates a `prototype.html` with inline CSS, JS, multi-screen routing. Open in browser. Better for iteration. |
| AsciiKit | Dedicated tool for building ASCII wireframes with Claude, ChatGPT, Cursor |
| MockFlow export | Draw wireframe visually; export as an optimised prompt for Claude Code |
| ui-wireframing skill (playbooks.com) | Claude Code skill that generates `WIREFRAME.md` files. Fills the text-based spec gap. |

### The Jane Street pattern

One Jane Street designer builds directly in Claude Code instead of Figma — iterating on working prototypes, presenting a functional feature instead of a design proposal. Key insight: all effort goes into the real artifact, none into "ancillary in-between work" like Figma components.

Works for solo/small team. Breaks down when collaborative design review is needed before build.

## The gap in Simon's workflow

Simon's four-gate product design process (documented in `product-design-skill-spec.md`) has no prototype step:

```
Gate 1 (Problem) → Gate 2 (Design spec) → Gate 3 (Implementation plan) → Build
```

The missing step is between Gate 2 and Gate 3:

```
Gate 2 (Design spec) → Gate 2.5 (Prototype + review) → Gate 3 (Implementation plan)
```

Without a prototype step, the implementation plan is written from a text description of the UI. Problems are discovered mid-build. The prototype step catches navigation errors, missing states, and layout problems before a line of implementation code is written.

## Decision: Option A — Annotated HTML prototype

Three options were evaluated:

| Option | What | Verdict |
|---|---|---|
| A — Annotated HTML prototype | Claude generates a single-file prototype with a click-to-annotate layer. Copy feedback back to Claude. | **Chosen** |
| B — Claude Artifacts | Fast loop in claude.ai, no annotation UI, no persistence | Good for quick iteration, not structured review |
| C — Figma | Full design review, most overhead | Worth it for stakeholder review; overkill for solo builds |

### What Option A produces

A single `.html` file containing:
- Full clickable multi-screen prototype of the feature
- An annotation layer — click any element to add a sticky note
- Notes panel listing all annotations (numbered)
- "Copy feedback" button — exports all notes as structured text to paste back into Claude

### The feedback loop

```
Design spec
→ Claude generates annotated HTML prototype
→ Simon opens in browser, clicks through flows, adds sticky notes
→ Copies structured feedback
→ Pastes to Claude
→ Claude updates design spec or feeds directly into implementation plan
→ Build
```

## What needs to be built

A Claude skill (or a recurring prompt pattern) that:

1. Takes a design spec (or section of one) as input
2. Generates a single-file annotated HTML prototype with the annotation layer
3. Opens it in the browser automatically

This does not need to be a formal stored skill immediately. It can start as a prompt pattern tested in a session. If it works well, it becomes a skill (`product-design` or a sub-step within the four-gate process).

**Minimum viable version:**
- Multi-screen HTML prototype matching the spec
- Click-to-annotate on any element
- Notes panel with numbered entries
- Copy feedback button (structured text output)

**Not needed in v1:**
- Persistence (notes don't need saving between sessions)
- Server or database
- Sharing / collaboration

## PRD vs Product Design Spec — clarification

A **PRD** is engineering-facing: user stories, requirements (P0/P1/P2), acceptance criteria, success metrics. Answers: "what does done look like?"

A **product design spec** is broader: what the product is, information architecture, data model, API, UI layout, visual design, write paths, roles, out of scope. The full design before requirements.

Simon's current documents (e.g., `2026-02-27-knowledge-base-redesign.md`) are product design specs, not PRDs. For solo builds, the design spec functions as the PRD — one document. Separate PRDs are only needed when handing off to an engineering team or when acceptance criteria need to be tracked independently.
